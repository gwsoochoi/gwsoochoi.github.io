# Refactor

`git diff origin/main` 기준으로 변경 코드를 분석하고, 리팩토링 포인트를 도출한다.

## 절차

### 1. diff 수집

```bash
git diff origin/main --name-only
git diff origin/main --stat
```

### 2. 코드 품질 분석

각 변경 파일을 다음 관점으로 체크한다:

| 관점 | 체크 항목 |
|------|----------|
| **DRY** | 동일 로직이 반복되는가? 컴포넌트 추출이 필요한가? |
| **일관성** | 사전 파일(ko/ja/en) 간 구조가 동기화되어 있는가? |
| **타입 안전** | Dictionary 인터페이스와 실제 데이터가 일치하는가? |
| **컴포넌트 분리** | 하나의 컴포넌트가 너무 많은 책임을 가지는가? |
| **Tailwind** | 반복되는 클래스를 추출할 수 있는가? |

### 3. 파일 끝 개행 검증

파일 끝 개행 누락 시 자동 추가하고 보고한다.

### 4. 보고

리팩토링 포인트를 번호와 함께 정리한다:

```
1. [코드 위치] 현재 문제 → 변경 제안 (Before/After 코드 블록 포함)
2. ...
```

- 포인트가 없으면 **"추가 리팩토링 포인트 없음"**을 보고한다.
- 있으면 사용자에게 번호 선택을 요청한다 (예: "1, 3 진행해"). 승인된 항목만 구현한다.
- 과도한 엔지니어링을 피하고, 실질적인 유지보수 이득이 있는 경우에만 제안한다.

# Audit

기존 기능의 코드·타입·비즈니스 규칙을 심층 분석하여, 잠재적 위험·취약점·엣지 케이스를 추론하고 구조화된 보고서를 출력한다.

**읽기 전용** — 파일을 절대 수정하지 않는다.

## 인자

```
/audit {feature}
```

- `feature`: 감사 대상 영역 — `resume`, `about`, `post`, `blog`, `layout`, `i18n`, `theme` 등

feature가 비어있으면 올바른 형식(`/audit {feature}`)을 안내하고 종료한다.

## 절차

### 1. 컨텍스트 로딩

feature에 해당하는 다음 파일을 탐색·파악한다:

| 소스 | 읽는 이유 |
|------|----------|
| `app/[locale]/*.tsx` / `app/[locale]/{feature}/` | 컴포넌트 구현 파악 |
| `lib/content/*.ts` | 데이터 구조 및 타입 파악 |
| `messages/{ko,ja,en}.json` | i18n 키 일관성 파악 |
| `app/globals.css` | CSS 변수 사용 현황 파악 |

### 2. 분석 파이프라인

3단계 파이프라인으로 분석한다. **코드 근거**가 있는 항목만 보고한다.

---

#### Phase 1: 구조 분석 — 잠재적 결함이 있는가?

##### R (Risk) — 잠재적 위험 요소

| 분석 대상 | 확인 항목 |
|-----------|----------|
| 타입 안전 | `any` 사용, 옵셔널 체이닝 누락, 타입 단언 남용 |
| 데이터 정합성 | `lib/content/*.ts`의 필드가 null/undefined일 때 렌더링 경로 |
| 빌드 의존성 | `generateStaticParams()` 누락으로 정적 빌드 실패 가능성 |
| i18n 키 | ko/ja/en 사전 간 키 불일치 또는 누락 |

##### V (Vulnerability) — 보안·논리적 취약점

| 분석 대상 | 확인 항목 |
|-----------|----------|
| XSS | `dangerouslySetInnerHTML` 사용 여부 |
| 민감 정보 | 환경 변수, API 키 하드코딩 |
| 접근 제어 | devOnly 페이지(`posts`)가 프로덕션 빌드에서 노출되는지 |

##### L (Limitation) — 현재 방식의 한계점

| 분석 대상 | 확인 항목 |
|-----------|----------|
| 정적 사이트 제약 | 동적 데이터 요구가 발생할 경우 현재 구조로 대응 불가한 부분 |
| 다국어 레이아웃 | ko/ja/en 텍스트 길이 차이로 인한 레이아웃 깨짐 가능성 |
| 테마 전환 | CSS 변수를 직접 색상값(`text-gray-500` 등)으로 우회한 경우 |

---

#### Phase 2: 운영 분석 — 실사용 환경에서 문제가 되는가?

##### B (Bottleneck) — 성능 병목 구간

| 분석 대상 | 확인 항목 |
|-----------|----------|
| 렌더링 | 불필요하게 큰 클라이언트 컴포넌트 (`"use client"` 남용) |
| 이미지 | `images.unoptimized: true` 환경에서 큰 이미지 미최적화 |
| 번들 크기 | 무거운 클라이언트 사이드 의존성 |

##### CN (Concerns) — 기술 부채 우려 사항

| 분석 대상 | 확인 항목 |
|-----------|----------|
| 코드 복잡도 | 컴포넌트 크기, 중첩 깊이, 하나의 컴포넌트에 너무 많은 책임 |
| 일관성 | Tailwind 직접 색상 사용 (`text-gray-*` 등) vs CSS 변수 혼용 |
| 번역 하드코딩 | UI 텍스트가 `useTranslations()` 없이 하드코딩된 경우 |

##### VAR (Variables) — 외부 조건 변화 시 영향받는 요소

| 분석 대상 | 확인 항목 |
|-----------|----------|
| CDN 폰트 | Pretendard CDN 장애 시 폴백 폰트 적용 여부 |
| GitHub Pages | 배포 경로 변경 시 static export 설정 영향 |
| Node.js 버전 | CI(Node 25)와 로컬 환경 불일치 가능성 |

---

#### Phase 3: 엣지 케이스 추론 — 예외 상황에서 정상 동작하는가?

Phase 1·2에서 발견한 항목을 기반으로 논리적으로 추론한다.

| 추론 소스 | 도출 방법 | 예시 |
|-----------|----------|------|
| 옵셔널 필드 | 해당 필드가 undefined일 때 렌더링 경로 추적 | `career.ts`의 `endDate`가 없을 때 |
| i18n 키 누락 | 사전에 키가 없을 때 next-intl 동작 | 신규 추가 키가 en.json에만 있을 때 |
| 테마 전환 | localStorage가 없는 환경(SSR) | 초기 렌더링 시 hydration mismatch |
| 빌드 타임 | devOnly 조건 분기 | NODE_ENV 환경변수 미설정 시 |
| 긴 텍스트 | 다국어 텍스트 길이 극단값 | 일본어 경력 타이틀이 매우 길 때 레이아웃 |

**추론 프로세스:**
1. Phase 1·2 발견 항목 각각에 대해 "이 상황이 극단적으로 발생하면?" 질문
2. 코드에서 해당 경로의 방어 로직(guard, optional chaining, fallback) 존재 여부 확인
3. 방어 로직이 **없거나 불충분**한 경우만 보고

---

### 3. 보고서 출력

#### 심각도 기준

| 심각도 | 기준 | 조치 |
|--------|------|------|
| 🔴 Critical | 빌드 실패·보안 사고·화면 깨짐 가능 | 즉시 수정 권장 |
| 🟡 Warning | 특정 조건에서 오동작 가능 | 계획적 수정 권장 |
| 🔵 Info | 개선하면 좋지만 당장 문제 없음 | 참고 |

#### 보고 규칙

- **코드 근거 없는 추측은 보고하지 않는다.** 파일:라인 참조 필수.
- 발견이 없는 관점은 "해당 없음"으로 표시하고 넘어간다.
- Phase 3 엣지 케이스는 **"방어됨"도 포함** — 현재 방어 상태를 가시화한다.
- 과도한 엔지니어링 제안을 피한다. 실질적 위험만 보고한다.

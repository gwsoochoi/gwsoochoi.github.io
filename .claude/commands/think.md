# Think

현재 코드베이스와 UI/UX 패턴을 분석하고 유저 요청에 대해 깊이 추론한 후 솔루션을 제안한다.

**읽기 전용** — 파일을 절대 수정하지 않는다.

## 인자

```
/think {feature}: {instruction}
```

- `feature`: 분석 대상 영역 — `resume`, `about`, `post`, `blog`, `layout`, `i18n` 등
- `instruction`: 고민하고 있는 내용

예시:
- `/think resume: 경력 섹션을 타임라인 형태로 바꾸면 어떨까`
- `/think blog: 마크다운 기반 블로그를 추가하는 방법`
- `/think layout: 다크모드 지원 전략`

## 절차

### 1. 현재 상태 파악

- 관련 코드, 컴포넌트, 사전 파일을 확인한다.
- 프로젝트의 기존 UI/UX 패턴 및 컨벤션을 파악한다.
- 제약사항을 확인한다 (Next.js 정적 사이트, GitHub Pages, 다국어 3개).

### 2. 요청 추론

- 컨텍스트에서 의도를 파악하고 모호한 부분을 명확화한다.
- 엣지 케이스, 의존성, 잠재적 충돌을 식별한다.
- 3개 언어(ko/ja/en) 모두에 미치는 영향을 평가한다.

### 3. 솔루션 제안

- 1~3개 옵션과 트레이드오프를 제시한다.
- 최적 옵션을 추천하고 근거를 설명한다.
- 구현 범위를 명시한다: 변경 파일, 신규 컴포넌트.

### 4. 확인 요청

- 추천 방향을 2~3문장으로 요약한다.
- 코드 작성 전 유저 확인을 요청한다.

## 출력 형식

### 현재 상태
[현재 무엇이 있는지]

### 추론
[파악한 의도 + 엣지 케이스 + 제약사항]

### 제안
**옵션 A** — [이름]: [설명] | 공수: S/M/L
**옵션 B** — [이름]: [설명] | 공수: S/M/L

### 추천
[최적 옵션 + 이유]

> 옵션 X로 진행할까요?

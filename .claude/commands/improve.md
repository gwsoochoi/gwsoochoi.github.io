# Improve

현재 구현된 기능의 UI/UX를 분석하여 더 높은 퀄리티로 개선한다.

`/refactor`가 코드 구조(DRY/일관성)에 집중한다면, `/improve`는 **사용자 경험과 표현 품질**에 집중한다.

## 인자

```
/improve {feature}: {instruction}
```

- `feature`: 개선 대상 영역 — `resume`, `about`, `post`, `blog`, `layout`, `i18n` 등
- `instruction`: 개선하고 싶은 내용

예시:
- `/improve resume: 경력 섹션 가독성 개선`
- `/improve layout: 전체 여백 및 타이포그래피 정리`
- `/improve i18n: 언어 전환 UX 개선`

## 절차

### 1. 현재 구현 분석

feature 영역의 코드, 스타일, 레이아웃을 파악한다.

### 2. 개선 포인트 도출

다음 관점에서 현재 구현을 평가하고 개선 포인트를 도출한다:

| # | 관점 | 체크 항목 |
|---|------|----------|
| 1 | 레이아웃 | 정보 계층, 여백, 정렬이 적절한가? |
| 2 | 타이포그래피 | 폰트 크기, 굵기, 줄간격이 가독성에 최적인가? |
| 3 | 반응형 | 모바일/태블릿/데스크톱에서 레이아웃이 자연스러운가? |
| 4 | 접근성 | 시맨틱 HTML, 색상 대비, 스크린 리더 지원이 적절한가? |
| 5 | SEO | meta 태그, 구조화 데이터, OG 이미지가 적절한가? |
| 6 | 다국어 | ko/ja/en 텍스트 길이 차이로 인한 레이아웃 깨짐이 없는가? |
| 7 | 인터랙션 | hover/focus 상태, 전환 효과가 적절한가? |

### 3. 개선안 보고

개선 포인트를 번호와 함께 정리한다:

```
1. [관점] 현재 상태 → 개선 제안
2. ...
```

- 포인트가 없으면 **"추가 개선 포인트 없음"**을 보고한다.
- 있으면 사용자에게 번호 선택을 요청한다 (예: "1, 3 진행해"). 승인된 항목만 구현한다.
- 과도한 엔지니어링을 피하고, 실질적인 사용자 가치가 있는 경우에만 제안한다.

### 4. 구현

승인된 개선 항목을 구현한다. 3개 언어 모두 영향을 확인한다.

### 5. 검증

```bash
npx tsc --noEmit
npx next build
```

### 6. 결과 보고

변경된 파일 목록과 개선 요약을 제시한다.
